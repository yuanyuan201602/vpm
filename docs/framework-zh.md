# Verified Project Management (VPM)

**AI agent 协作项目的目标层可信化框架。**

VPM 位于 Harness Engineering 和 Loop Engineering 之上。后两者问的是 *"agent 怎么运行？"*，VPM 问的是 *"agent 在向正确的目标运行吗——它到达终点时会知道吗？"*

---

## 用一个数据点说明问题

一个真实项目。同一个开发者，同一套 Claude + Codex 技术栈：

```
npm test          → 54 个测试通过   ✓
npm run typecheck → 通过            ✓
npm run build     → 通过            ✓
npm run sim:user  → 12/12           ✓

npm run test:experience → 0/8       ✗
```

所有指标绿灯。产品完全偏离目标。

工程健康线一直在通过。产品价值线——唯一真正在问"我们做出了 PRD 承诺的东西吗"的那条线——一直在失败，而且从来没有人赋予它叫停项目的权力。

这不是执行失败。agent 执行得非常好。它只是在向错误的目标执行。

## VPM 增加了什么

三个层次，嵌套关系：

```
┌─────────────────────────────────────────────┐
│  Verified Project Management (VPM)          │
│  "目标正确吗？仍然正确吗？验收线              │
│   对准了真实价值吗？"                        │
│                                             │
│  ┌───────────────────────────────────────┐  │
│  │  Loop Engineering                    │  │
│  │  "循环会收敛吗？                     │  │
│  │   什么时候停止？"                    │  │
│  │                                      │  │
│  │  ┌────────────────────────────────┐  │  │
│  │  │  Harness Engineering          │  │  │
│  │  │  "agent 需要什么运行环境？"   │  │  │
│  │  └────────────────────────────────┘  │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

VPM 不替代另外两层。它治理它们所指向的目标。

## 五条规则

**1. PRD 和验收标准是同一个产物，不是两个。**
它们共享版本号，一起变更。如果你写不出产品价值线检查项，PRD 就还没写完。

**2. 验收分两层。只有第二层是闸门。**
- 工程健康线：能跑、能构建、通过烟测。必要条件，不充分。
- 产品价值线：是否实现了 PRD 承诺的价值。这是唯一能叫停项目的东西。

**3. PRD 变更是最危险的时刻。**
PRD 变更时：冻结旧方向（标记 legacy），以新版本重新推导验收标准，重新推导里程碑。绝不在旧验收标准下继续。

**4. 核心先于外壳。**
在核心功能通过产品价值线之前，不做打包、桌面封装、分享功能或品牌设计。每一个过早的外围工作都感觉像进展，实则在证明错误的方向。

**5. 确定性事实用确定性检查。判断类问题用独立审阅。**
禁用词、构建成功、字节对齐——grep 和运行脚本。年龄适宜性、身份完整度、叙事连贯性——独立审阅者读真实产物，不读生产者的自我报告。

## 与其他框架的关系

| 框架 | 问的问题 | 范围 |
|------|---------|------|
| Harness Engineering | agent 需要什么运行环境？ | 单 agent，单任务 |
| Loop Engineering | 循环如何收敛，何时停止？ | 单循环，已定义目标 |
| **VPM** | **目标正确吗？验收对准了真实价值吗？** | **完整项目生命周期，目标漂移，PRD 版本管理** |

---

## 与行为规则集的关系（卡帕西 12 条）

VPM **不替代**卡帕西 CLAUDE.md 这类单任务行为契约——它叠在上面。

那些规则让每一个 agent 工作单元安全:写代码前先读上下文、主动暴露失败、测意图而非行为、做手术式修改。它们是必要的。但 agent 可以完美遵守全部 12 条,仍然造出错误的产品——每个任务都成功,项目却整体漂离目标。这个缺口正是 VPM 要补的。

VPM 的安装套件把这些规则作为 `for-coders/AGENTS.md` 的**行为层**,置于它自己的**治理层**之下。完整的层级映射、VPM 借鉴了哪些、又刻意拒绝了哪条,见 [docs/related-work.md](related-work.md)。
